# Contributing to tree-sitter-crystal

## Development Setup

```bash
npm install
npm run generate   # Generate parser from grammar.js
npm run test       # Run the test corpus (197 tests)
```

## Project Architecture

```
grammar.js            ← Crystal syntax rules (edit this)
src/scanner.c         ← External scanner for strings, regex, heredocs, etc.
src/parser.c          ← Auto-generated by `tree-sitter generate` — DO NOT EDIT
src/grammar.json      ← Auto-generated — DO NOT EDIT
queries/
  highlights.scm      ← Syntax highlighting for editors
  tags.scm            ← Code navigation (symbol outline)
  locals.scm          ← Variable scoping (rename, references)
test/corpus/*.txt     ← Test cases (input + expected S-expression trees)
examples/*.cr         ← Sample Crystal files for manual testing
```

### How It Works

1. **grammar.js** defines Crystal's syntax as declarative rules using tree-sitter's DSL
2. **src/scanner.c** handles tokens that need stateful, context-aware lexing (strings with interpolation, heredocs with terminators, regex vs division, percent literals with balanced delimiters)
3. `tree-sitter generate` compiles both into `src/parser.c`
4. The parser produces a concrete syntax tree (CST) for any `.cr` file
5. Query files (`.scm`) map CST nodes to editor features

### Key Design Decisions

- **Macro `{{ }}` is opaque**: Content inside `{{ }}` is not parsed as Crystal. The token participates in expressions (dot calls, scoped constants, arguments) but its interior is treated as raw text.
- **Macro `{% %}` is statement-level**: Control flow tags are opaque tokens at the statement level only.
- **No-paren calls are greedy**: `foo bar` parses as a call, which can cause ambiguity with binary expressions. This is a known limitation addressed with `prec.left(-1)` on `argument_list_no_parens`.
- **External scanner uses a context stack**: Nested interpolation (e.g., `"#{a + "#{b}"}"`) is handled by pushing/popping context entries.

## Making Changes

### Modifying the Grammar

1. Edit `grammar.js`
2. Run `npm run generate` to regenerate the parser
3. Run `npm run test` to verify all tests pass
4. Add test cases for new syntax in `test/corpus/`

### Modifying the Scanner

1. Edit `src/scanner.c`
2. Run `npm run generate` (also rebuilds the scanner)
3. Run `npm run test`
4. If adding new external tokens, update BOTH the `externals` array in `grammar.js` AND the `TokenType` enum in `src/scanner.c` — order must match

### Adding Tests

Tests use tree-sitter's corpus format in `test/corpus/*.txt`:

```
==================
Test name
==================

crystal source code here

---

(source_file
  (expected_s_expression_tree))
```

Tips:
- Use `npx tree-sitter parse file.cr` to see the actual parse tree
- Supertype nodes (like `expression`) are hidden in test output
- Named fields appear as `field_name: (node)` in the S-expression
- Run a single test file: `npx tree-sitter test --filter "test name"`

### Benchmarking Against Crystal Stdlib

```bash
# Download stdlib files
curl -sL "https://raw.githubusercontent.com/crystal-lang/crystal/master/src/array.cr" > /tmp/crystal_array.cr

# Parse and count errors
npx tree-sitter parse /tmp/crystal_array.cr 2>&1 | grep -c "ERROR\|MISSING"
```

## File Conventions

| Pattern | Meaning |
|---------|---------|
| Rules starting with `_` | Hidden from AST (e.g., `_body`, `_method_name`) |
| `token()` | Single terminal node, no internal structure |
| `token.immediate()` | No whitespace allowed before this token |
| `prec()` / `prec.left()` / `prec.right()` | Precedence and associativity |
| `field('name', ...)` | Named child accessor in the AST |
| `PREC.*` constants | Operator precedence levels (higher = tighter) |
